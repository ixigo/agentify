Install once. Your coding agent uses it automatically.

Agentify gives AI coding agents lightweight, persistent context. Run agentify install once per repo — from then on the agent drives Agentify, not you.

Why Agentify over a raw agent?

Raw Claude Code and Codex start every session amnesiac: they re-derive project knowledge, retry known dead ends, run full test suites, and review their own code with their own blind spots. Agentify fixes each of those with plumbing, not prompting. Every terminal output below is captured from a real run in a small payment-service demo repo — the review findings are Codex's actual words, the costs are the provider's actual numbers.

1 · It remembers what failed — and says so before the retry

Command results are tracked, not just commands. When the agent is about to rerun something that failed in a previous session, a PreToolUse hook injects a warning first:

# session 1: the agent runs a migration; it fails. sessions later, it tries again —
Agentify: this exact command failed in a previous session (2026-07-07) (exit 1):
error: relation "orders" already exists. If the underlying cause was not fixed
since, try a different approach instead of retrying it as-is.

One later success clears the failure. Warnings dedupe per session and never fire for failures the current session already saw.

2 · It runs the tests your change affects — not the whole suite

agentify test walks the import graph in the structural index from your changed files to the test files that actually exercise them:

$ agentify test   # only src/pay/retry.js was modified
Selected tests: 1 file(s) from 1 changed file(s)
- test/retry.test.js (imports changed code)
Run:
- node --test test/retry.test.js

The unrelated test/dash.test.js is skipped. --run executes the selection and exits non-zero on failure; when no related tests exist, it says so and recommends the full suite — silence is never treated as coverage.

3 · A different vendor reviews the code — and catches real things

The review route goes to the other vendor by default. This is Codex's unedited verdict on a commit that added ts: Date.now() to a retry payload:

$ agentify review --diff HEAD~1
cross-vendor review by codex — diff since HEAD~1

**Findings**
- Medium: src/pay/retry.js now adds `ts: Date.now()` to the retry return value.
  This makes retryPayment(2, "k1") non-deterministic and changes the public shape
  from { attempt, key } to { attempt, key, ts }. For payment retry/idempotency
  code, that is risky: callers that serialize, compare, dedupe, cache, or sign
  the retry payload can now see different values for the same attempt/key.

Opt in with hooks.prePush: true and this runs on every git push — advisory, never blocking.

4 · Its memory flags its own rot

A note that references a file which no longer exists is the most dangerous kind of memory — confidently wrong. Notes are verified against the working tree at injection time:

### Notes left for this session
- [2026-07-07] payment idempotency key logic lives in src/pay/retry.js — never regenerate per attempt
- [2026-07-07] legacy card vault code is in src/pay/vault-legacy.js, do not touch without compliance
  — STALE? references missing path(s): src/pay/vault-legacy.js; verify before trusting

5 · Decisions stay decided

Settled trade-offs get relitigated by every new session unless they're on record — and queryable:

$ agentify ctx decisions "why client side idempotency keys"
Decisions matching "why client side idempotency keys":
- [2026-07-07] chose idempotency keys stored client-side over server-side dedup
  because the gateway retries before our server sees the request

With agentify ctx share, the committed notes file doubles as a lightweight team ADR log.

6 · You can see what the routing costs

Every delegation is logged with the provider's real usage numbers where the CLI reports them. Actual output after one Haiku research call and one Codex review:

$ agentify stats
Delegations:
- total: 2 run(s), 50.1k in / 530 out, cost $0.0212 (1/2 reported)

By kind:
- research: 1 run(s), 49.9k in / 301 out, cost $0.0212
- review:   1 run(s), 199 in / 229 out, cost n/a

Note: token counts for 1 run(s) are estimates (~4 chars/token); the provider
CLI reported no usage.

7 · It works beyond Claude Code and Codex

Everything above is also exposed as an MCP server — Cursor, Zed, Windsurf, Claude Desktop, Gemini CLI, anything MCP-capable:

$ agentify serve   # stdio MCP server, zero extra dependencies
tools: ctx_load, ctx_note, ctx_match, query, risk, test_select

8 · It's benchmarked against plain Claude Code — and reported honestly

An 8-task paired benchmark runs in Harbor (Terminal-Bench 2.0) containers: same image, same pinned model, same budget, same deterministic verifier — the only difference between the arms is agentify install plus context recorded in earlier sessions. First full nightly suite, 2026-07-14, three attempts per arm:

agentify      24/24 passed   $0.055/attempt   Wilson 95% CI  86.2–100%
claude-code   21/24 passed   $0.033/attempt   Wilson 95% CI  69.0–95.7%

All 3 baseline failures: avoid-cache-regression — the task where a recorded
production incident is the only difference between the arms (0/3 vs 3/3).
Controls held: the mechanical task and the misleading-note task tied 3/3.
Flakes: 0 in 48 trials. Total spend: $2.10 against a $16.80 hard ceiling.

Verdict: no winner declared — 3 discordant pairs give sign-test p = 0.25;
the fail-closed rule requires CI separation AND p < 0.05. That bar is the
point: the report never manufactures a confident winner from thin data.

About that cost gap: persistent context is paid for in tokens, so the context arm runs ~66% more per attempt — and on the only task with signal, that ~$0.03 premium was the difference between failing and passing. This suite isolates the context layer only. In real workflows the model routing in #6 is designed to win the premium back — quick and mechanical work goes to economy models, reviews to the other vendor — but that's an economics claim this container benchmark deliberately does not measure; agentify stats and agentify value report what routing actually saves from your own logged runs, and the optimization-profile suite exists to test it.

How it works

One command wires Agentify into your agent's configuration, then stays out of your way.

agentify install
  ├── CLAUDE.md             ← managed guidance block: how the agent should use agentify
  ├── .claude/settings.json ← Claude Code hooks:
  │     SessionStart     → agentify ctx load     (inject context digest)
  │     UserPromptSubmit → agentify ctx match    (inject only task-relevant context)
  │     PreToolUse       → agentify ctx precheck (warn before repeating a failed command)
  │     PostToolUse      → agentify ctx track    (record edits, commands + failures)
  │     SessionEnd       → agentify ctx track    (close out the session)
  └── .agentify/            ← lightweight JSONL context store + structural index
01

You ask for something — each prompt is matched against the store and only related notes and files are injected (set context.injection: digest for the old always-on digest).

02

Agent works — file edits and shell commands are tracked automatically as compact JSONL (auto-compacted, capped at ~512 KB).

03

Agent learns something worth keeping — it runs agentify ctx note "..." so the knowledge survives the session.

04

Session ends — a fast model compresses it into a ~3-line handoff for future sessions; agentify ctx share makes notes committable team memory.

No daemon, no database server, no per-command wrapping. Install and uninstall are surgical: only the marker-delimited block and Agentify's own hook entries are ever touched.

Quick start

# install the CLI straight from GitHub
curl -fsSL https://raw.githubusercontent.com/ixigo/agentify/main/install.sh | bash
# (equivalent: npm install -g github:ixigo/agentify)

# wire up a repo (CLAUDE.md + Claude Code hooks)
cd /path/to/your/repo
agentify install

# also works with Codex (writes AGENTS.md guidance)
agentify install --provider codex
agentify install --provider all

# or one global setup instead of per-repo files
agentify install --global

# optional: build the structural index for query/risk
agentify scan

# shell work out to the right model
agentify delegate quick "rename getUser to fetchUser" --write
agentify delegate review --diff origin/main

# check or undo at any time
agentify status
agentify uninstall

Works with

Claude Code automatic

Hooks track every file edit and command with zero agent effort. A SessionStart hook injects the context digest into each new session. Guidance lives in CLAUDE.md.

Codex guidance-driven

Codex has no lifecycle hooks, so agentify install --provider codex writes an AGENTS.md block teaching Codex to load context at session start and record notes as it works.

Any other agent CLI

Everything is a plain CLI with --json output: ctx load, ctx note, query, risk. Add equivalent guidance to your agent's instruction file.

Platform workflows

On GitHub, GitLab, or Azure DevOps — there's a prebuilt workflow to get things done. agentify workflow install detects your platform from the git remote and installs the bundle: triage the board, pick up an item, implement in an isolated worktree, raise a draft PR.

GitHub gh

github-triage label state machine, gh-autopilot execution, issue-killer parallel worktree fan-out, draft PRs via pr-creator.

GitLab glab

gitlab-triage, glab-autopilot for issues and MRs, the same worktree isolation and parallel fan-out, draft MRs.

Azure DevOps az

azure-devops-triage over Azure Boards, ado-autopilot for work items and Repos PRs, plus pr-convention-learner for reviewer conventions.

agentify workflow install   # auto-detects gh / glab / azure from the git remote
agentify workflow list      # bundles + platform CLI availability

You never invoke the workflow by name — the skills match plain requests. A typical day:

"triage the new issues"          → github-triage labels the board, marks agentify-ready
"pick up issue 231"              → isolated worktree, implement, test, cross-vendor review, draft PR
"kill everything agentify-ready" → issue-killer fans out parallel tmux worktree agents
next session                     → the SessionStart digest recalls what's in flight

Model routing

Install also sets up a routing table: each kind of work goes to the model best suited for it, and the agent is taught to delegate instead of doing everything inline.

KindDefault routeUsed for
quickClaude HaikuSmall, low-impact edits, mechanical changes, quick questions
implementClaude SonnetStandard feature work and multi-file refactors
heavyClaude OpusArchitecture decisions, deep debugging, high-risk changes
reviewCodexIndependent post-change review by a different vendor
researchClaude HaikuFast exploration, summarization, doc lookups
# after finishing a change, the agent gets a second opinion from another vendor
agentify delegate review --diff origin/main

**Findings**
- **High**: math.js:1 changes add(a, b) from addition to subtraction.
  Callers expecting add(2, 1) === 3 now get 1 — a direct behavioral regression.

Routes use version-independent model aliases so they don't rot, fall back across vendors when a CLI is missing, run read-only unless --write is passed, and are fully overridable in .agentify.yaml under models.routes.

What the agent sees

Context arrives when it's relevant. Ask about payment retries, and only the related notes and files are injected — a CSS question pulls nothing:

## Agentify context (relevant to this task)

### Related notes from earlier sessions
- [2026-07-07] payment idempotency key logic lives in src/pay/retry.js
  — never regenerate per attempt

### Related commands that previously failed (avoid repeating as-is)
- [2026-07-07] `npm run migrate` (exit 1) — error: relation "orders" already exists

### Files previously worked on that look related
- src/pay/retry.js (14 edits)
- test/retry.test.js (9 edits)

Full history: `agentify ctx load`.

Matching is BM25-ranked with light stemming — "payment retries" finds the retry module's notes, rare terms outrank boilerplate, and injected items dedupe per session. Events are single-line JSON records under .agentify/context/. Command text is clipped, output is never captured, and secrets are redacted from run reports.

Design principles

Lightweight

Context is plain JSONL. The event log auto-compacts past ~512 KB. Hook commands are designed to never fail and never block the agent.

Reversible

Everything install writes is marker-delimited or prefix-identified. uninstall removes exactly what was added; your own config survives.

Agent-first

JSON output everywhere, structural queries over a deterministic index, and guidance blocks that teach the agent when to reach for each command.
