Repository Intelligence Layer

How Agentify works inside a codebase

Agentify is not another coding model. It is the orchestration layer around agent-driven work: it scans a repository, writes a structured SQLite index, plans bounded context for a task, wraps external provider CLIs, refreshes documentation after changes, validates safety rules, and preserves session continuity.

Ad hoc context

What this project actually is

A CLI that coordinates repository indexing, task planning, provider execution, documentation generation, validation, cleanup, and resumable sessions for AI-assisted engineering work.

Entry
src/cli.js forwards commands into src/main.js.
Index

Planner

Overview

What problem it solves and why the design matters

The main issue with AI coding in real repositories is not code generation itself. The real issues are context sprawl, stale repo knowledge, vendor lock-in, and the lack of a durable project memory. Agentify addresses those by turning the repository into a structured system the agent can query and refresh.

Key idea: orchestration over improvisation

Without a layer like this

  • An agent scans too broadly or too narrowly because every prompt starts from scratch.
  • Repository docs go stale the moment the code changes.
  • One provider's CLI conventions leak into the whole workflow.
  • Long tasks lose continuity between sessions and branches.
  • The repo accumulates metadata and run artifacts with no lifecycle.

With Agentify in the middle

  • The repo is indexed once into a reusable knowledge base, then queried for each task.
  • Prompt context is selected from modules, files, symbols, tests, and commands instead of by guesswork.
  • Provider execution is abstracted, so the workflow stays stable even if the model vendor changes.
  • Sessions preserve bootstrap context and checklist state for resume and fork flows.
  • Cleanup, cache management, and validation keep generated artifacts from rotting.
Workflow

How the project works from command to checked result

The system is easiest to understand as a staged pipeline. The important detail is that each stage produces state the next stage can trust, rather than passing around an unstructured prompt.

1

Initialization

agentify init writes baseline artifacts and config so the repo has a standard place for index, docs, runs, cache, and session state.

  • Repo-local defaults live in .agentify.yaml.
  • Sticky provider preference is persisted per repository, not globally.
2

Deterministic scan and index

scan walks the repository, detects stacks and modules, fingerprints files, and writes a SQLite index.

  • TypeScript and JavaScript get AST-based symbol and import extraction.
  • Python, .NET, Java, Kotlin, Go, Rust, and Swift are also detected, with more generic import and symbol heuristics.
  • The scan also identifies tests, config files, entrypoints, key files, and inferred runnable commands.
3

Bounded planning for a task

plan and run use the index to score modules, files, symbols, changed files, related tests, and verification commands against the task text.

  • The planner works under byte budgets, so it can hand providers only the most relevant excerpts.
  • Dependency edges boost neighboring modules when a strong match is found.
4

Provider execution

Agentify builds a provider-specific command for Codex, Claude, Gemini, or OpenCode, or wraps a custom command via exec.

  • run is the opinionated path that uses the planner prompt.
  • exec is the lower-level wrapper if you want your own command after --.
5

Automatic refresh after changes

After an execution finishes, Agentify compares pre and post git snapshots. If changes were made, it refreshes scan and docs, then validates the repository.

  • This is the part that keeps metadata synchronized with code instead of leaving it stale.
  • Full up also runs project tests and writes an HTML run report.
6

Session continuity and maintenance

sess run, sess resume, and sess fork store context and bootstrap state under .agents/session/. Cleanup and cache commands keep this state from growing forever.

  • Context and bootstrap are clipped to byte limits to stay practical for agent handoff.
  • Cleanup prunes stale docs, dead sessions, old run reports, and orphaned cache blobs.
Architecture

What each major subsystem does, how it works, and why it helps

These are the core moving parts in the repo. Each one exists to convert an unreliable manual workflow into something queryable, repeatable, and maintainable.

Detection Indexer Multi-language

1. Stack and module detection

The detection layer inspects repo signals such as package.json, workspace files, Gradle files, Swift package structure, Go modules, Rust manifests, and language extensions to determine the most likely stack and module layout.

  • What: infer stacks and module boundaries without hardcoding one framework.
  • How: file-system heuristics plus workspace-aware logic, especially for TypeScript monorepos.
  • Why it helps: the planner and docs operate on modules, so large repos stop looking like one flat file pile.
SQLite Structured data

2. Repository index database

The index lives in .agents/index.db and stores repo metadata, modules, files, symbols, imports, tests, commands, artifacts, and index events. It runs with WAL mode and foreign keys enabled.

  • What: a stable, queryable memory of the repo.
  • How: scans write normalized rows into tables rather than serializing one giant JSON blob.
  • Why it helps: owner lookup, dependency queries, change tracking, and planning become cheap and deterministic.
Planner Budgets

3. Task-aware execution planning

The planner tokenizes the task, scores modules, files, and symbols, applies dependency boosts, reads bounded excerpts, and outputs a focused prompt with relevant tests and verification commands.

  • What: turn a user task into a constrained context packet.
  • How: weighted scoring from names, paths, symbols, changed files, entrypoints, key files, and tests.
  • Why it helps: better prompts with less noise means less wandering, fewer broad repo scans, and cheaper execution.
Provider adapters CLI wrapping

4. Provider abstraction

Provider commands are mapped centrally. Codex, Claude, Gemini, and OpenCode each get a command template, while local remains valid for deterministic scan, doc, and check flows.

  • What: separate repo orchestration from model vendor choice.
  • How: build provider-specific command arrays from one common interface.
  • Why it helps: teams can switch providers without rewriting the whole repo workflow.
Documentation Caching

5. Documentation and metadata generation

The doc phase prepares repo context, optionally asks an external provider for a manager plan and per-module artifacts, caches by fingerprint, writes module docs, updates key-file headers, and produces AGENTIFY.md.

  • What: durable documentation tied to indexed repo state.
  • How: fingerprinted module contexts, cache hits, concurrency, and structured provider output.
  • Why it helps: code understanding becomes an asset in the repo, not a one-time chat transcript.
Validation Guardrails

6. Freshness and safety checks

Validation checks that the stored index matches the current git head and that generated changes stay within allowed paths. For code files, it compares content with the HEAD version after stripping Agentify headers, so it can detect body changes.

  • What: fail-closed guardrails around generated repo state.
  • How: git diffs, allowlisted paths, header-only comparison, and index freshness checks.
  • Why it helps: the tool is opinionated about metadata safety instead of pretending all automated output is trustworthy.
Sessions Cleanup Cache

7. Continuity and hygiene

Sessions package bootstrap context, checklist state, and lineage so work can be resumed or forked. Cleanup removes orphaned docs, stale run reports, invalid sessions, ghost runs, and unreferenced cache blobs.

  • What: lifecycle management for agent-generated state.
  • How: retained-file pruning, invalid-directory detection, content-addressed blobs, and age-based GC.
  • Why it helps: long-running adoption stays manageable instead of turning into hidden repo clutter.

Index and query surface

  • .agents/index.db is the operational source of truth for modules, files, symbols, imports, tests, commands, and artifacts.
  • agentify query owner|deps|changed|search reads directly from that index.
  • This is what makes the tool useful for real navigation and planning, not just documentation.

Human-readable repo documentation

  • AGENTS.md and docs/repo-map.md summarize structure after scan.
  • AGENTIFY.md and docs/modules/*.md capture module-level understanding and start-here guidance.
  • This converts agent reasoning into documentation another engineer can inspect later.

Runs, reports, and ghost mode

  • Run reports store token usage, module counts, validation status, and test status.
  • agentify-report.html is a rendered snapshot of one run's health.
  • Ghost mode writes under .current_session/ so you can inspect planned output without touching the main artifact path.

SQLite tables that matter

  • repo_meta stores root facts such as detected stack and head commit.
  • modules, files, symbols, and imports represent the structural graph.
  • tests and commands connect code to runnable verification.
  • artifacts caches reusable outputs by fingerprint.
  • index_events records scan history details.

What makes the docs generation stronger

  • There is a repo-level manager plan plus per-module artifacts instead of one flat summary.
  • Module context is fingerprinted, so unchanged modules can be reused from cache.
  • The provider output is expected to match a schema, and fallback artifacts exist when external generation fails.
Commands

How the command surface is organized

The command set is split between repository preparation, execution, continuity, querying, and maintenance. That matters because it keeps routine work separate from deeper orchestration concerns.

Command What it does When to use it Why it matters
init Create baseline Agentify artifacts and config. First-time setup in a repo. Standardizes the operating layout.
scan / index Build the repository index and repo map. Before planning, querying, or doc generation. Turns the repo into structured data.
doc Generate module docs, metadata, and headers. When you want repo memory and navigable docs. Keeps knowledge in the repo, not only in chats.
plan Preview the selected modules, files, symbols, tests, and commands for a task. Before running an agent when you want to inspect scope. Gives visibility into context routing.
run Build a planner prompt, run a provider template command, then refresh and validate. Default path for task execution. Combines planning and post-change hygiene.
exec Wrap any command after -- and optionally refresh afterward. Advanced usage with custom provider commands. Keeps refresh logic even when you customize execution.
sess Create, resume, list, or fork session state. Long tasks, alternate branches, or handoff workflows. Adds continuity and lineage.
query Answer owner, dependency, changed-module, and search questions from the index. When you need routing information fast. Makes the index operational, not decorative.
check Validate freshness and safety rules. Before trusting generated artifacts. Prevents silent drift.
up Run scan, doc, validation, and tests as a full pipeline. When you want a complete refreshed snapshot. Gives one reproducible repo refresh path.
clean / cache Prune stale outputs and inspect cache health. Ongoing maintenance. Stops long-term artifact sprawl.

Typical workflow

agentify init
agentify up --provider codex
agentify run "implement retry logic for checkout"
agentify sess run --provider codex --name "payments-v2" "continue implementation"
agentify query owner --file src/core/planner.js
agentify clean

Important operational detail

  • Sticky provider: explicit --provider updates the repo-local default.
  • Interactive mode: currently the template interactive flow is specific to the Codex provider.
  • Run versus exec: run chooses context for you; exec assumes you already know the command you want.
  • Up versus doc: up is the full pipeline; doc is the documentation and metadata slice.
Benefits

Why this is better for almost any codebase

The strongest case for Agentify is not that it writes docs or launches an agent. The strongest case is that it gives a codebase a reusable operating model for AI work: structured context, queryability, continuity, and maintenance.

Pitch: repo knowledge becomes infrastructure

Large or modular repos

Module detection, dependency mapping, owner lookup, and ranked context are immediately useful once the repo is bigger than a handful of files.

  • Better task routing.
  • Lower prompt noise.
  • Clearer start-here guidance.

Teams using more than one provider

Agentify keeps the workflow stable while the provider command changes. That avoids rewriting internal habits every time the preferred model changes.

  • Less vendor lock-in.
  • Stable operational commands.
  • Provider choice becomes a config detail.

Repos that need durable docs

Most agent sessions produce transient understanding. Agentify pushes that understanding back into the repo as index-backed docs and metadata.

  • Better onboarding.
  • Faster change reviews.
  • Shared understanding across sessions.

Long-running implementation work

Session bootstrap, checklist carryover, and forkable session IDs give the work memory beyond one chat window.

  • Resume where you left off.
  • Try alternate designs safely.
  • Keep a lineage of attempts.

Safety-conscious engineering orgs

Freshness validation, allowlisted generated paths, and conservative file-change rules create friction in the right place: before stale or unsafe artifacts are trusted.

  • Better auditability.
  • Clear failure categories.
  • Less silent metadata drift.

Any repo that accumulates AI output

Once a team starts generating indexes, docs, reports, and sessions, lifecycle management becomes necessary. Cleanup and cache GC make adoption sustainable.

  • Less artifact clutter.
  • Controlled retention.
  • More predictable disk use.

Best argument to engineering leadership

  • It reduces the hidden cost of repeated repo re-discovery for every AI task.
  • It makes agent work observable through index data, docs, reports, and validation.
  • It creates one internal workflow that can survive changes in model vendors.

Best argument to individual developers

  • You get faster navigation, better prompts, and resumable context.
  • You do less manual explanation of the repo on each task.
  • You can inspect why the planner chose certain files instead of trusting a black box.
Adoption

How to explain it internally without overselling it

The strongest explanation is honest: Agentify does not magically make every agent smarter. It makes the repo easier for agents and humans to work in by giving them structure, memory, and guardrails.

For platform teams

  • Standardize one repository operating model for AI workflows.
  • Keep generated state under named directories and retention rules.
  • Make provider switching a policy choice rather than a repo rewrite.

For leads and reviewers

  • Use the planner and docs as a first-pass view into what changed and why.
  • Use validation as a freshness and safety gate before trusting generated artifacts.
  • Use sessions to preserve decision context across long implementation threads.

For contributors

  • Run plan before a complex task to inspect scope.
  • Use run for normal task execution and sess for continuity.
  • Use query when you need ownership or dependency answers quickly.

Where it is strongest

  • TypeScript and JavaScript repos, especially workspaces and monorepos.
  • Teams that want multi-provider support without workflow churn.
  • Codebases large enough that repo understanding needs to be stored, not rediscovered.

Where the story should stay honest

  • It is an orchestration tool, not a replacement for tests, review, or engineering judgment.
  • Non-TypeScript stacks are supported, but the deepest semantic extraction is in the TS/JS path.
  • The validation model is intentionally strict, so teams need to adopt it with clear expectations.
FAQ

Direct answers to the questions people usually ask

Is Agentify itself an AI agent?
No. It is the orchestration layer around agent use. The codebase analysis, indexing, planning, session handling, validation, and artifact lifecycle are Agentify's job. The actual coding model is an external provider such as Codex, Claude, Gemini, or OpenCode.
Why not just call the provider CLI directly?
You can, but then you lose the index, planner-selected context, structured docs, repo-local provider preference, auto-refresh, validation, sessions, cache reuse, and cleanup workflow. Direct CLI execution is usually the shortest path; Agentify is the repeatable path.
Does it work only for JavaScript or TypeScript?
No. The repo shows support for TypeScript/JavaScript, Python, .NET, Java, Kotlin, Go, Rust, and Swift. The TypeScript path is deeper because it uses the TypeScript compiler APIs for imports and symbols. Other stacks still benefit from module detection, indexing, tests, commands, sessions, and cleanup.
What is the best reason to adopt it in a mature codebase?
Mature codebases already have the main failure modes this tool targets: too much context, too many modules, stale docs, repeated rediscovery work, and mixed human plus AI contributions. Agentify turns those into first-class repo state that can be refreshed, queried, and pruned.
What is the most realistic limitation?
It does not remove the need for engineering review. It improves context, structure, and continuity around AI work. The quality of the final code still depends on the provider, the tests, the task framing, and the team's standards.
