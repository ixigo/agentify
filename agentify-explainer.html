v1 · repository intelligence layer

orchestration for agent-driven codework.

agentify is not another coding model. it is the orchestration layer around agent use — it indexes the repo into SQLite, plans bounded context for a task, wraps external provider CLIs, refreshes documentation after changes, validates safety rules, and preserves session continuity.

agentify@repo — zsh 
$ agentify --explain
# repository intelligence layer — what happens on every task

→ scan     # walk repo · detect stacks · index modules, files, symbols, imports, tests
→ plan     # score modules · select bounded context under byte budgets
→ execute  # wrap codex | claude | gemini | opencode CLI with the planner prompt
→ refresh  # re-scan · regenerate docs · update headers after code changes
→ validate # freshness + allowlist + header-only guardrails
→ persist  # sessions · checklist state · forkable lineage

▶01/overview

what problem it solves

the real issues are context sprawl, stale repo knowledge, vendor lock-in, and the lack of durable project memory — not code generation itself. agentify turns the repo into a structured system the agent can query and refresh.

× without a layer like this

  • agent scans too broadly or narrowly — every prompt starts from scratch.
  • repo docs go stale the moment the code changes.
  • one provider's CLI conventions leak into the whole workflow.
  • long tasks lose continuity between sessions and branches.
  • the repo accumulates metadata and run artifacts with no lifecycle.

→ with agentify in the middle

  • the repo is indexed once into a reusable knowledge base, then queried per task.
  • prompt context is selected from modules, files, symbols, tests, and commands — not guesswork.
  • provider execution is abstracted; the workflow stays stable across vendor changes.
  • sessions preserve bootstrap context and checklist state for resume and fork flows.
  • cleanup, cache management, and validation keep generated artifacts from rotting.
▶02/workflow

command → result pipeline

each stage produces state the next stage can trust, rather than passing around an unstructured prompt.

  1. 01stage

    initialization

    agentify init writes baseline artifacts and config so the repo has a standard place for index, docs, runs, cache, and session state.

    $agentify init
    • repo-local defaults live in .agentify.yaml.
    • sticky provider preference is persisted per repository, not globally.
  2. 02stage

    deterministic scan and index

    agentify scan walks the repository, detects stacks and modules, fingerprints files, and writes a SQLite index.

    $agentify scan
    • typescript and javascript get AST-based symbol and import extraction.
    • python, .NET, java, kotlin, go, rust, and swift are also detected, with generic import and symbol heuristics.
    • the scan identifies tests, config files, entrypoints, key files, and inferred runnable commands.
  3. 03stage

    bounded planning for a task

    plan and run use the index to score modules, files, symbols, changed files, related tests, and verification commands against the task text.

    $agentify plan "refactor checkout"
    • the planner works under byte budgets, so providers receive only the most relevant excerpts.
    • dependency edges boost neighboring modules when a strong match is found.
  4. 04stage

    provider execution

    agentify builds a provider-specific command for codex, claude, gemini, or opencode — or wraps a custom command via exec.

    $agentify run "add retry logic" --provider codex
    • run is the opinionated path that uses the planner prompt.
    • exec is the lower-level wrapper when you want your own command after --.
  5. 05stage

    automatic refresh after changes

    once an execution finishes, agentify compares pre and post git snapshots. if changes were made, it refreshes scan and docs, then validates the repository.

    $agentify up
    • this is the part that keeps metadata synchronized with code instead of leaving it stale.
    • full up also runs project tests and writes an HTML run report.
  6. 06stage

    session continuity and maintenance

    sess run, sess resume, and sess fork store context and bootstrap state under .agents/session/. cleanup and cache commands keep this state from growing forever.

    $agentify sess resume <id>
    • context and bootstrap are clipped to byte limits to stay practical for agent handoff.
    • cleanup prunes stale docs, dead sessions, old run reports, and orphaned cache blobs.
▶03/architecture

moving parts

each subsystem converts an unreliable manual workflow into something queryable, repeatable, and maintainable.

01stack & module detection
detectionindexermulti-lang

the detection layer inspects repo signals such as package.json, workspace files, gradle files, swift package structure, go modules, rust manifests, and language extensions to determine the most likely stack and module layout.

→ why: the planner and docs operate on modules, so large repos stop looking like one flat file pile.

02repository index db
sqlitestructured data

lives in .agents/index.db and stores repo metadata, modules, files, symbols, imports, tests, commands, artifacts, and index events — with WAL mode and foreign keys enabled.

→ why: owner lookup, dependency queries, change tracking, and planning become cheap and deterministic.

03task-aware planning
plannerbudgets

tokenizes the task, scores modules, files, and symbols, applies dependency boosts, reads bounded excerpts, and outputs a focused prompt with relevant tests and verification commands.

→ why: better prompts with less noise means less wandering, fewer broad repo scans, and cheaper execution.

04provider abstraction
adapterscli wrapping

provider commands are mapped centrally. codex, claude, gemini, and opencode each get a command template; local remains valid for deterministic scan, doc, and check flows.

→ why: teams can switch providers without rewriting the whole repo workflow.

05docs & metadata generation
documentationcaching

the doc phase prepares repo context, optionally asks an external provider for a manager plan and per-module artifacts, caches by fingerprint, writes module docs, updates key-file headers, and produces AGENTIFY.md.

→ why: code understanding becomes an asset in the repo, not a one-time chat transcript.

06freshness & safety checks
validationguardrails

validation checks that the stored index matches the current git head and that generated changes stay within allowed paths. for code files, it compares content with the HEAD version after stripping agentify headers to detect body changes.

→ why: the tool is opinionated about metadata safety rather than pretending all automated output is trustworthy.

07continuity & hygiene
sessionscleanupcache

sessions package bootstrap context, checklist state, and lineage so work can be resumed or forked. cleanup removes orphaned docs, stale run reports, invalid sessions, ghost runs, and unreferenced cache blobs.

→ why: long-running adoption stays manageable instead of turning into hidden repo clutter.

index and query surface

.agents/index.db is the operational source of truth for modules, files, symbols, imports, tests, commands, and artifacts. agentify query owner|deps|changed|search reads directly from it — which is what makes the tool useful for real navigation, not just documentation.

human-readable docs

AGENTS.md and docs/repo-map.md summarize structure after scan. AGENTIFY.md and docs/modules/*.md capture module-level understanding and start-here guidance. agent reasoning becomes documentation another engineer can inspect later.

runs, reports, ghost mode

run reports store token usage, module counts, validation status, and test status. agentify-report.html is a rendered snapshot of one run's health. ghost mode writes under .current_session/ so you can inspect planned output without touching the main artifact path.

sqlite tables that matter

  • repo_meta — root facts such as detected stack and head commit.
  • modules, files, symbols, imports — the structural graph.
  • tests, commands — code connected to runnable verification.
  • artifacts — reusable outputs cached by fingerprint.
  • index_events — scan history details.

what makes docs generation stronger

  • a repo-level manager plan plus per-module artifacts, not one flat summary.
  • module context is fingerprinted, so unchanged modules are reused from cache.
  • provider output is expected to match a schema, with fallback artifacts when external generation fails.
▶05/commands

the command surface

split between preparation, execution, continuity, and maintenance — routine work is separate from orchestration concerns.

command what when why
setup
init create baseline agentify artifacts and config. first-time setup in a repo. standardizes the operating layout.
scan · index build the repository index and repo map. before planning, querying, or doc generation. turns the repo into structured data.
doc generate module docs, metadata, and headers. when you want repo memory and navigable docs. keeps knowledge in the repo, not only in chats.
execute
plan preview the modules, files, symbols, tests, and commands selected for a task. before running an agent when you want to inspect scope. gives visibility into context routing.
run build a planner prompt, run a provider template command, then refresh and validate. default path for task execution. combines planning and post-change hygiene.
exec wrap any command after -- and optionally refresh afterward. advanced usage with custom provider commands. keeps refresh logic even when you customize execution.
continuity
sess create, resume, list, or fork session state. long tasks, alternate branches, or handoff workflows. adds continuity and lineage.
query answer owner, dependency, changed-module, and search questions from the index. when you need routing information fast. makes the index operational, not decorative.
maintenance
check validate freshness and safety rules. before trusting generated artifacts. prevents silent drift.
up run scan, doc, validation, and tests as a full pipeline. when you want a complete refreshed snapshot. one reproducible repo refresh path.
clean · cache prune stale outputs and inspect cache health. ongoing maintenance. stops long-term artifact sprawl.
typical session 
$ agentify init
$ agentify up --provider codex               # full pipeline, sticky provider
$ agentify run "implement retry logic for checkout"
$ agentify sess run --provider codex \
    --name "payments-v2" "continue implementation"
$ agentify query owner --file src/core/planner.js
$ agentify clean                              # prune stale runs, sessions, blobs

operational notes

  • sticky provider. explicit --provider updates the repo-local default.
  • interactive mode. the template interactive flow is currently specific to the codex provider.
  • run vs exec. run chooses context for you; exec assumes you know the command already.
  • up vs doc. up is the full pipeline; doc is the docs and metadata slice.
▶06/benefits

why this helps almost any codebase

the strongest case is not that it writes docs or launches an agent — it is that it gives a codebase a reusable operating model for AI work: structured context, queryability, continuity, and maintenance.

large or modular repos

module detection, dependency mapping, owner lookup, and ranked context are immediately useful once the repo is bigger than a handful of files.

  • better task routing
  • lower prompt noise
  • clearer start-here guidance

multi-provider teams

keeps the workflow stable while the provider command changes — no rewriting internal habits every time the preferred model shifts.

  • less vendor lock-in
  • stable operational commands
  • provider choice becomes config

repos that need durable docs

most agent sessions produce transient understanding. agentify pushes that understanding back into the repo as index-backed docs and metadata.

  • better onboarding
  • faster change reviews
  • shared understanding across sessions

long-running implementation work

session bootstrap, checklist carryover, and forkable session IDs give work memory beyond a single chat window.

  • resume where you left off
  • try alternate designs safely
  • keep a lineage of attempts

safety-conscious orgs

freshness validation, allowlisted generated paths, and conservative file-change rules create friction in the right place — before stale or unsafe artifacts are trusted.

  • better auditability
  • clear failure categories
  • less silent metadata drift

repos accumulating AI output

once a team starts generating indexes, docs, reports, and sessions, lifecycle management becomes necessary. cleanup and cache GC make adoption sustainable.

  • less artifact clutter
  • controlled retention
  • predictable disk use

repo knowledge becomes infrastructure. agent work becomes observable through index data, docs, reports, and validation — and one internal workflow survives changes in model vendors. individual engineers get faster navigation, better prompts, resumable context, and can inspect why the planner chose certain files instead of trusting a black box.

— the argument for engineering leadership and individual developers
▶07/adoption

how to explain it without overselling

the honest pitch: agentify does not magically make every agent smarter. it makes the repo easier for agents and humans to work in by giving them structure, memory, and guardrails.

01for platform teams
  • standardize one repository operating model for AI workflows.
  • keep generated state under named directories and retention rules.
  • make provider switching a policy choice rather than a repo rewrite.
02for leads & reviewers
  • use the planner and docs as a first-pass view into what changed and why.
  • use validation as a freshness and safety gate before trusting generated artifacts.
  • use sessions to preserve decision context across long implementation threads.
03for contributors
  • run plan before a complex task to inspect scope.
  • use run for normal execution and sess for continuity.
  • use query when you need ownership or dependency answers fast.

where it is strongest

  • typescript and javascript repos — especially workspaces and monorepos.
  • teams that want multi-provider support without workflow churn.
  • codebases large enough that repo understanding must be stored, not rediscovered.

where the story stays honest

  • it is an orchestration tool, not a replacement for tests, review, or engineering judgment.
  • non-typescript stacks are supported, but the deepest semantic extraction is in the ts/js path.
  • the validation model is intentionally strict — adopt it with clear expectations.
▶08/faq

direct answers

is agentify itself an AI agent?
no. it is the orchestration layer around agent use. the codebase analysis, indexing, planning, session handling, validation, and artifact lifecycle are agentify's job. the actual coding model is an external provider such as codex, claude, gemini, or opencode.
why not just call the provider CLI directly?
you can, but then you lose the index, planner-selected context, structured docs, repo-local provider preference, auto-refresh, validation, sessions, cache reuse, and cleanup workflow. direct CLI execution is the shortest path; agentify is the repeatable path.
does it work only for javascript or typescript?
no. the repo shows support for typescript/javascript, python, .NET, java, kotlin, go, rust, and swift. the typescript path is deeper because it uses the typescript compiler APIs for imports and symbols. other stacks still benefit from module detection, indexing, tests, commands, sessions, and cleanup.
what is the best reason to adopt it in a mature codebase?
mature codebases already have the main failure modes this tool targets — too much context, too many modules, stale docs, repeated rediscovery work, and mixed human plus AI contributions. agentify turns those into first-class repo state that can be refreshed, queried, and pruned.
what is the most realistic limitation?
it does not remove the need for engineering review. it improves context, structure, and continuity around AI work. the final code still depends on the provider, the tests, the task framing, and the team's standards.
