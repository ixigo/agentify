Setup is a sequence of checkpoints.

Each command has a narrow job. Run the macOS bootstrap when you want help preparing the machine, run manual initialization when the tools are already managed elsewhere, then keep the repo fresh with refresh and validation checkpoints.

decision points

Know which doorway you are opening.

The close pairs are intentionally different. Pick the command whose behavior matches the current checkpoint instead of treating them as aliases.

this vs init

agentify this is the guided macOS path. It requires a non-local provider, checks Homebrew and provider readiness, can install missing bootstrap tools after confirmation, and then initializes the repo. agentify init only writes the Agentify baseline for the target repo.

up vs check

agentify up changes generated state by running scan, optional docs, validation, and detected tests. agentify check is the validation checkpoint: it reports whether freshness, schemas, and safety rules still match the repo.

sync timing

agentify sync is for an already-Agentified repo after the CLI itself changes. It refreshes repo-owned config, baseline artifacts, managed hooks, and repo-scoped built-in skills, then runs maintenance with the local provider.

checkpoints

Six commands cover the setup slice.

Each card lists the purpose, when to use it, key flags from the current CLI surface, and a realistic command you can run without a build step.

inspect

doctor

Purpose: Checks setup readiness, provider CLIs, auth where supported, package-manager readiness, optional memory tooling, and capability tier.

Use when: You switch machines, debug tooling, or want a read-only report before setup.

Key flags: --semantic, --json, --fail-on-stale.

agentify doctor --semantic
guided setup

this

Purpose: Bootstraps the current macOS repo for a provider-backed Agentify workflow.

Use when: You want Agentify to verify/install local bootstrap tools after confirmation and prepare the repo in one guided path.

Key flags: --provider, --root, --dry-run, --json.

agentify this --provider codex --root /path/to/repo
manual setup

init

Purpose: Creates baseline Agentify artifacts such as config, ignore files, guardrails, runtime folders, and module-doc fallback folders.

Use when: You manage system dependencies yourself, work outside macOS bootstrap, or want a minimal repo initialization.

Key flags: --provider, --root, --dry-run, --json.

agentify init --provider codex
refresh

up

Purpose: Runs scan, optional docs, validation, and detected project tests as the normal maintenance pipeline.

Use when: The repo changed and you want the index, docs, validation, and test signal refreshed together.

Key flags: --provider, --docs, --hook, --strict, --provider-timeout-ms.

agentify up --provider local --docs=false
validate

check

Purpose: Validates freshness, schemas, generated state, and safety rules without running the full refresh pipeline.

Use when: You are about to commit, validating hook behavior, or confirming generated artifacts after a refresh.

Key flags: --hook, --strict, --json, --root.

agentify check --hook
upgrade

sync

Purpose: Upgrades repo-owned Agentify files and then runs the maintenance pipeline with the deterministic local provider.

Use when: The Agentify CLI gained new repo-level files, defaults, managed hooks, or built-in project skills.

Key flags: --provider, --docs, --dry-run, --json.

agentify sync --provider codex --dry-run
blocked states

Failure states are signals.

These commands report blocked or failing checkpoints instead of promising automatic repair. Use the message to choose the next command or install step.

missing provider this rejects local and needs a provider such as codex, claude, gemini, or opencode. In non-interactive mode, pass --provider.
unsupported platform this currently supports macOS. On Linux, CI, or pre-provisioned machines, use init, install missing tools yourself, then rerun doctor.
stale artifacts check can report stale generated state. Run up when you want Agentify to refresh scan/docs/check/test outputs.
intentional edits Use check --hook or up --hook when hook-friendly validation should skip source body diffing after intentional code changes.
doc provider timeout Use --provider-timeout-ms for provider-backed docs, or choose --provider local when you want deterministic maintenance.
upgrade drift Run sync after upgrading Agentify so existing repos adopt new baseline files, managed hooks, and repo-scoped skills.

After the setup route passes, start provider work with agentify run or save a durable workflow with agentify sess.
