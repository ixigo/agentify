Sessions turn a provider-backed task into durable campaign state. Start a run, resume later with the same context, fork from an earlier path, or hand the next agent a compact bundle.

mental model

A session is a campaign checkpoint.

The command wraps provider launches with local state. Agentify records enough structure for the next launch to know where the work started, what context was selected, and what artifacts were generated.

save file

agentify sess run creates .agentify/session/<id>/, a manifest, bounded context, a checklist, and optional markdown bootstrap files before launching the provider.

load game

agentify sess resume rehydrates an existing session by ID, rebuilds the provider prompt from saved bootstrap/context, and appends the new task to the same workstream.

new route

agentify sess fork starts a child route from a parent session, carrying forward selected checklist and run history while keeping the new session path separate.

inventory

agentify sess list shows known sessions with ID, provider, and creation time, so completions and handoffs can target the right run.

moves

The four session moves cover the loop.

Use these commands when one provider turn is not enough and you want Agentify-managed continuity instead of relying only on provider history.

start

Run a new session

Creates a fresh session unless --session points at an existing one. Use --name for a readable label and --provider to pin the agent host.

agentify sess run --provider codex --name "payments-v2"
continue

Resume by ID

Loads session-manifest.json, context.json, and bootstrap.md. If no task is provided, the provider is instructed to continue from the latest repository state.

agentify sess resume --session <session-id> "finish the tests"
branch

Fork from a parent

Creates a separate child session from --from. This is useful when a workstream splits into an experiment, review path, or handoff branch.

agentify sess fork --from <session-id> --name "payments-v2-review"
inspect

List saved sessions

Prints recent session IDs, provider names, and creation timestamps. With JSON output enabled, the command returns the full manifest list.

agentify sess list

How a session moves through state.

The session flow is deliberately file-backed and local. Nothing becomes durable unless Agentify writes it into the session directory.

Artifacts under .agentify/session/<id>/

The manifest names the session and provider. The context files keep the launch bounded. Runtime records capture what happened after the provider starts.

session-manifest.json Session ID, parent/fork IDs, provider, name, creation time, index snapshot, and cache refs.
context.json Fitted snapshot with module IDs, checklist summary, run history, rolling summary, and artifact refs.
checklist.json Structured task checklist carried from a parent session when the new run is forked.
bootstrap.md Markdown launch brief generated from the bounded context when markdown artifacts are enabled.
transcript.md Readable run transcript used by automatic session memory and future recall.
memory-context.md Automatic memory excerpt attached to the launch when prior session memory is available.
launches.jsonl Provider launch records, including the redacted command, provider, capture mode, and prompt metadata.
turns.jsonl Structured turn records used for replay, recall, and managed-context estimates.
context-events.jsonl Context fetch and routing events emitted during the session workflow.
context-facts.json Compacted structured facts extracted from session state when context compaction runs.
context-facts.md Markdown version of compacted context facts for humans and provider prompts.
handoff.json / handoff.md Cross-agent bundle written by agentify handoff --session <id>.

Session IDs are validated before path resolution, and session directories are kept under .agentify/session/.
