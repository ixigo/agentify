afk issue execution

Turn issues into agent-ready work.

AFK issue work is not a bigger prompt. It is a narrow GitHub issue, a durable agent brief, an opt-in label or explicit URL, supervised tmux panes, and a draft PR with checks the maintainer can inspect.

prepare issue launch panes
default label agentify-ready session gh-issue-killer providers codex or claude
supervised queue github only v1 
# Preview labelled AFK issues first
agentify issue-killer --issue-provider github --label agentify-ready --agent-provider codex --limit 2 --dry-run

# Launch selected panes when the queue is ready
agentify issue-killer --issue-provider github --label agentify-ready --agent-provider codex --limit 2

# Supervise the agents
tmux attach -t gh-issue-killer
classification

AFK work is narrow, testable, and unblocked.

Before applying the label, decide whether the issue can be implemented from written context alone. Anything that needs a product call, design review, ambiguous tradeoff, or human approval before code should stay HITL.

afk

Ready for autonomous implementation

The desired behavior is explicit, acceptance criteria are verifiable, scope boundaries are written down, and the agent can run tests or checks without waiting for a human answer.

labels: agentify-ready
hitl

Needs human judgment first

The issue asks for a direction decision, has unclear UX or API behavior, depends on secrets or live access, or cannot define done without a reviewer clarifying intent.

keep unlabelled until resolved
issue contract

Write the issue so an agent can start cold.

The issue body and agent brief are the operating contract. Prefer behavior, interfaces, evidence, acceptance checks, and out-of-scope boundaries over step-by-step file edits that can drift.

problem

State the current behavior, the desired behavior, and the user or maintainer impact. Include concrete evidence such as command output, failing tests, screenshots, or source references that explain why the issue exists.

brief

Add an agent brief or equivalent acceptance contract. Name key interfaces and expected behavior, then list independently testable acceptance criteria. Avoid brittle line-number instructions.

scope

Call out what must not change. This prevents agents from picking adjacent cleanup, broad refactors, unrelated issues, or speculative feature work while they are AFK.

verify

List the smallest useful checks: focused unit tests, browser inspection for HTML pages, CLI help comparison, git diff --check, or the repo test command when shared behavior is touched.

launch path

Use opt-in labels or explicit issue URLs.

The v1 launcher is intentionally selective. It does not scan arbitrary backlog items. It launches GitHub issues chosen by the configured label or by exact issue URLs.

label queue

Launch from the opt-in queue

Use this when several issues are already reviewed and labelled. The default label is agentify-ready, the default limit is 5, and supervised v1 runs cap --limit at 10.

agentify issue-killer --issue-provider github --label agentify-ready --agent-provider codex --limit 5
exact urls

Launch only named issues

Use explicit URLs when you want a hand-picked set. With URLs, the launcher defaults the limit to the number of URLs, so unrelated labelled issues stay untouched.

agentify issue-killer --issue-url https://github.com/ORG/REPO/issues/123,https://github.com/ORG/REPO/issues/124 --agent-provider claude
prerequisites

Check readiness before showing launch commands.

Dry runs require repository and GitHub access. Real pane launches also require worktree orchestration, tmux, and the selected agent provider on PATH.

git The launcher verifies the current checkout is a Git repository before selecting issues or creating worktrees.
gh auth status GitHub auth is required for issue listing, issue views, branch context, and draft PR creation by agents.
wt Real launches use Worktrunk to create one isolated worktree per selected issue branch.
tmux Each selected issue runs in an interactive supervised pane, with the default session named gh-issue-killer.
codex / claude V1 agent providers are codex and claude. Choose one with --agent-provider.
permission prompts Provider sandbox and permission prompts remain active unless --bypass-permissions is explicitly passed.
guardrails

Supervised does not mean unguarded.

AFK launches should reduce prompting, not remove review. Keep the selection explicit, keep pane prompts scoped to their assigned issue, and use permission bypass only when the operator intentionally accepts that mode.

selection

Use --label or --issue-url. Do not turn the whole backlog into a launch queue, and do not add the opt-in label until the issue is truly AFK-ready.

limits

Start small. The default limit is 5; v1 rejects values above 10 so the tmux session stays observable.

bypass

--bypass-permissions is an explicit opt-in for provider permission bypass. It should not be the default example or the first command a new operator runs.

supervision

Attach to tmux, watch for blocked panes, stop misleading work, and expect each agent to keep shell, git, package-manager, test, commit, push, and draft PR actions scoped to its assigned issue.

completion checks

AFK is done when the draft PR is reviewable.

The agent should leave a Conventional Commit, push its issue branch with upstream tracking, and create a draft PR that links the source issue. The maintainer still checks the evidence before merging.

inspect output

Review the PR, not just the pane

Confirm the draft PR links the issue, names the touched behavior, separates unrelated failures, and reports the exact tests or browser checks run.

gh pr view <number> --json title,isDraft,body,headRefName
local proof

Re-run focused checks when needed

Use the issue's verification checklist as the first source of truth. For static HTML guides, open desktop and mobile widths and run whitespace checks.

git diff --check
scope control

Reject unrelated cleanup

The branch should solve the assigned issue only. Ask for follow-up issues instead of merging opportunistic refactors or adjacent docs churn.

git diff --stat origin/main...
blocked panes

Prefer honest blockers

If prerequisites, auth, tests, or product ambiguity block completion, the pane should leave a concise blocker note and skip misleading PR creation.

tmux attach -t gh-issue-killer
