# GitLab Issue Triage

Triage issues in the current GitLab repo using `glab`. Infer the repo from
`git remote` and confirm it with `glab repo view` before any mutation.

## AI Disclaimer

Every comment or issue posted to GitLab during triage must include this
disclaimer at the top of the body, before any other content:

```markdown
> *This was generated by AI during triage.*
```

## Prerequisites

Before reading or mutating GitLab state, run:

```bash
git rev-parse --is-inside-work-tree
command -v glab
glab auth status
glab repo view
GLAB_USER="$(glab api user | jq -r .username)"
```

If `jq` is unavailable, use `glab api user` and read the `username` field from
the JSON output.

## Labels

GitLab projects often have existing label taxonomies. Prefer existing labels
over creating new ones. If an equivalent label is missing, report it and ask
before creating it.

Recommended category labels:

| Label | Type | Description |
| --- | --- | --- |
| `bug` | Category | Something is broken |
| `enhancement` | Category | New feature or improvement |

Recommended state labels:

| Label | Type | Description |
| --- | --- | --- |
| `needs-triage` | State | Maintainer needs to evaluate this issue |
| `needs-info` | State | Waiting on reporter for more information |
| `ready-for-agent` | State | Fully specified, ready for agent work |
| `ready-for-human` | State | Requires human implementation |
| `wontfix` | State | Will not be actioned |

Every issue should have at most one state label and one category label from the
project's chosen taxonomy. If labels conflict, flag the conflict and recommend
one state before doing anything else.

## State Machine

| Current State | Can transition to | Who triggers it | What happens |
| --- | --- | --- | --- |
| `unlabeled` | `needs-triage` | Skill | Issue needs maintainer evaluation. Skill applies the label after presenting a recommendation. |
| `unlabeled` | `ready-for-agent` | Maintainer | Issue is already well specified and agent-suitable. Skill writes an agent brief comment, applies the label. |
| `unlabeled` | `ready-for-human` | Maintainer | Issue requires human implementation. Skill writes a brief summary comment, applies the label. |
| `unlabeled` | `wontfix` | Maintainer | Issue is duplicate, spam, or out of scope. Skill comments and closes only when explicitly instructed. |
| `needs-triage` | `needs-info` | Maintainer | Skill posts triage notes capturing progress so far and specific questions. |
| `needs-triage` | `ready-for-agent` | Maintainer | Skill writes an agent brief comment, applies the label. |
| `needs-triage` | `ready-for-human` | Maintainer | Skill writes a summary comment and applies the label. |
| `needs-triage` | `wontfix` | Maintainer | Skill comments and closes only when explicitly instructed. |
| `needs-info` | `needs-triage` | Skill | Reporter has replied. Skill surfaces the issue for maintainer re-evaluation. |

The maintainer can override the state directly, but unusual transitions should
be called out before mutation.

## Workflow: Show What Needs Attention

When the maintainer asks for an overview, query GitLab and present a grouped
summary:

1. Unlabeled issues: no category or state labels.
2. `needs-triage` issues.
3. `needs-info` issues with new activity since the last triage note.

Show counts per group. Within each group, show issues oldest first with issue
number, title, age, labels, and a one-line body summary.

Useful read-only commands:

```bash
glab issue list --state opened --per-page 100 --output json
glab issue view <number> --output json
glab issue note list <number>
```

If `glab issue note list` is unavailable in the installed `glab` version, use
`glab api` for the project's issue notes endpoint and state the fallback used.

## Workflow: Triage A Specific Issue

### Step 1: Gather Context

Before presenting a recommendation:

- Read the full issue body, labels, author, assignees, state, and timestamps.
- Read all comments or notes.
- Parse prior triage notes from earlier sessions.
- Inspect relevant code or docs to understand the domain and likely behavior.
- Check existing project labels before recommending new labels.

### Step 2: Present A Recommendation

Tell the maintainer:

- Category recommendation: bug or enhancement, mapped to the project label if
  it exists.
- State recommendation with reasoning.
- Any label conflicts or missing labels.
- Prior triage context and reporter replies since the last triage note.
- Brief codebase context relevant to the issue.

Wait for maintainer direction before mutating labels, comments, or state unless
the request was an explicit quick override.

### Step 3: Bug Reproduction

For bugs, attempt to reproduce before recommending `ready-for-agent`:

- Read reproduction steps.
- Trace relevant code paths.
- Run a focused command or test when practical.
- Report confirmed behavior, failed reproduction, or missing information.

### Step 4: Apply The Outcome

Depending on the maintainer's decision:

- `ready-for-agent`: post an agent brief comment and apply the state label.
- `ready-for-human`: post a summary comment and apply the state label.
- `needs-info`: post triage notes with what is known and specific questions.
- `needs-triage`: apply the label, optionally with a progress comment.
- `wontfix`: comment and close only when the maintainer explicitly asks.

Use conservative label changes: add the chosen state label and remove only
conflicting state labels from the same taxonomy.

## Agent Brief Template

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug / enhancement
**Summary:** One-line description of the work.

**Current behavior:**
Describe what happens now.

**Desired behavior:**
Describe what should happen, including edge cases and errors.

**Key interfaces:**
- Interface or contract that should change and why

**Acceptance criteria:**
- [ ] Specific, testable criterion
- [ ] Specific, testable criterion

**Out of scope:**
- Adjacent work that should not be included
```

## Needs Info Template

```markdown
> *This was generated by AI during triage.*

## Triage Notes

**What we've established so far:**

- Point 1
- Point 2

**What we still need from you (@reporter):**

- Specific question 1
- Specific question 2
```

## Guardrails

- Do not create labels unless the maintainer explicitly approves.
- Do not close issues unless explicitly requested.
- Do not mutate unrelated issues or merge requests.
- Do not apply GitHub-only label names blindly when the GitLab project already
  uses another taxonomy.
- Do not post comments without the AI disclaimer.
